## Three planes plus Git

| Plane | Use | Does not do |
|---|---|---|
| Local Collab 0.3 | Atomic same-device tasks, reports, review, checkpoints, and refusable handoffs over one SQLite file | Replicate between devices |
| Agent Correspondence 0.1 | Signed project-private facts and durable replay across devices | Choose a claim winner or transfer bytes |
| Collab release room 0.4 | Atomic cooperative external-operation lease and bounded provider evidence | Grant provider authority or stop direct provider actions |
| Git | Commits, branches, fetch, review, and merge | Coordinate release-side effects |

Use Correspondence for intent, advisory claims, progress, Git-addressed
artifacts, acknowledgements, conflict, refusal, rest, handoff, and repair.
Keep overlapping claims visible. Use the release room only immediately before
an external mutation.

## Explicit enrollment and credential boundary

Run the host-side `agenttool-collab-enroll` command once per device and
repository. Do not infer enrollment from a clone, remote URL, readable
profile, or human-facing label.

The wrapper uses an existing project bearer only for
`POST /v1/collab/enrolments`. On first enrollment it obtains a fresh
repository-scoped `atc_` bearer (generated for Keychain or pre-generated by a
scoped environment wrapper); re-enrollment reuses the existing device bearer
and does not rotate it. The wrapper derives the profile SHA-256 and canonical
allowed provider list and submits only that non-secret policy plus the token
prefix and SHA-256 digest. The relay stores the allowlist on the enrolled
device and enforces it for custom clients; `atc_` possession alone cannot
enable Vercel or another omitted provider.
Enrollment uses a deterministic exact-request key and device-version CAS. The
private metadata file persists the hash-only pending request before HTTP and
reuses it after an ambiguous response instead of rolling state backward.
An exact retry may return a durable observation receipt from before a later
policy change; this historical replay appends nothing and enables nothing.

Without an explicit credential path or device ID, the host uses one stable
repository-scoped `default.json` in its state directory and stores the
generated UUID inside it. A private local per-credential enrollment lock spans
the metadata read, HTTP request, and final version-fenced write. It prevents
competing local processes from replacing metadata; it is not a remote lock.

Supply the one-shot project bearer through
`AGENTOOL_COLLAB_PROJECT_BEARER` or the explicit
`--project-bearer-stdin` mode, never argv. Do not expose either bearer to the
model.

On macOS, keep the raw scoped bearer in Keychain. The selected mode-`0600`
metadata file contains only relay/repository/device metadata and the Keychain
reference:

```text
AGENTOOL_COLLAB_RELAY_CREDENTIAL_FILE=/host/private/metadata.json
AGENTOOL_COLLAB_RELAY_URL=https://relay.example
```

For a bounded CI/non-macOS process, provide
`AGENTOOL_COLLAB_RELAY_TOKEN` through a scoped wrapper and let its environment
end with the command. Never export it in global shell startup. Repository
requests present the scoped bearer over TLS; the relay is server-readable and
not end-to-end encrypted.

## Operation tools and lifecycle

Read with:

```text
collab_operation_events
collab_operation_status
collab_provider_list
```

Mutate with:

```text
collab_operation_claim
collab_operation_renew
collab_operation_begin
collab_operation_complete
collab_operation_release
collab_operation_recover
collab_provider_observe
```

Expect these tools only when the host configured a relay URL, validated
project profile, and active scoped credential binding. Plugin startup does not
enroll a device. With no relay URL the local tools remain available; a partial
or mismatched relay configuration fails startup rather than silently
discarding the remote coordination boundary.

The slot key is:

```text
repository + operation + environment
```

For this repository, cooperating clients use these exact pairs:

| Side effect | Operation | Environment |
|---|---|---|
| Branch push/delete | `github-branch` | `repository` |
| Pull-request create/update/close | `github-pull-request` | `repository` |
| Merge to the release branch | `github-merge` | `main` |
| `collab-v<version>` tag plus protected npm workflow | `npm-release` | `production` |
| Fly/Cloudflare release wrapper | `production-deploy` | `production` |
| Vercel preview deployment in an enabled project | `vercel-deploy` | `preview` |
| Vercel production deploy/promote in an enabled project | `vercel-deploy` | `production` |

Do not substitute names such as `publish`, `deploy-prod`, or `pr`; clients that
choose different strings acquire different slots. Another project may define
another vocabulary, but all of its cooperating clients must use the same
committed profile/runbook.

Bind each action to an exact target, Git source revision, and canonical
parameter SHA-256. The host should call the package's exported
`requestSha256(parameters)` over one agreed JSON object. It recursively sorts
object keys, preserves array order, and hashes the UTF-8 `JSON.stringify`
bytes with SHA-256. Do not calculate it manually or include credentials,
volatile timestamps, logs, or command output. The canonical npm-release
parameters use `trusted` because `@agenttool/collab` already exists; bootstrap
was restricted to its completed first publication:

```json
{
  "authentication": "trusted",
  "npm_tag": "<latest|next>",
  "package": "collab",
  "tag": "collab-v<version>",
  "workflow": "publish-npm.yml",
  "workflow_dispatch": {
    "inputs": {
      "authentication": "trusted",
      "npm_tag": "<same approved npm_tag>",
      "package": "collab",
      "tag": "collab-v<version>"
    },
    "ref": "collab-v<version>"
  }
}
```

Use these exact target/parameter profiles. Angle-bracket text is replaced with
the concrete value, `|` lists allowed alternatives rather than literal text,
and a safe integer is a JSON number rather than a quoted string. Fields shown
as `null` stay present as JSON null so every device hashes the same shape:

- `github-branch / repository`: target `refs/heads/<branch>`; parameters
  `{ "action": "push|delete", "expected_old_revision": "<hex>|null",
  "force": false, "new_revision": "<hex>|null" }`.
- `github-pull-request / repository`: for create, target
   `head:<head-ref>->base:<base-ref>`; otherwise target `pull:<decimal>`.
  Parameters are `{ "action": "create|update|close", "base_ref": "<ref>",
  "head_ref": "<ref>", "metadata_sha256": "<hex>|null",
  "pull_number": "<safe integer>|null" }`.
- `github-merge / main`: target `pull:<decimal>`; parameters
  `{ "expected_head_revision": "<hex>", "merge_method":
  "merge|squash|rebase", "release_branch": "main" }`.
- `npm-release / production`: target `@agenttool/collab@<version>`;
  parameters are exactly the six-key npm object above.
- `production-deploy / production`: target
  `surfaces:<comma-separated profile keys in lexical order>`; parameters
  `{ "receipt_schema": "agenttool-deploy-receipt/v2", "script":
  "bin/deploy.sh", "surfaces": ["<profile keys in lexical order>"] }`.
- `vercel-deploy / preview|production`: target
  `team:<team-id>/project:<project-id>`; parameters
  `{ "action": "deploy|promote|redeploy", "deployment_id": "<id>|null",
  "git_ref": "<ref>", "project_id": "<stable id>", "team_id":
  "<stable id>" }`.

Every GitHub branch value, including PR `head_ref` and `base_ref`, is the full
`refs/heads/<branch-name>` accepted by `git check-ref-format`. Do not use a
short name, remote-tracking name, or GitHub `owner:branch` label. This profile
supports PR branches in the bound repository; stop for a committed profile
extension rather than improvising a fork ref.

For a PR create or title/body update, compute `metadata_sha256` as
`requestSha256({ body: approvedBody, title: approvedTitle })`. Both keys are
present and describe the complete resulting metadata: `title` is the exact
approved string and `body` is the exact approved string or JSON null. Preserve
Unicode and line endings without trimming or normalization. Use
`metadata_sha256: null` when an update changes neither field or the action is
`close`; never upload the title/body to the room.

For npm, take `<version>` exactly from
`bun bin/npm-release.ts resolve --package collab`, without a leading `v`, and
use it in target `@agenttool/collab@<version>`. Derive
`release_tag = "collab-v" + version`, then use `release_tag` for the annotated
tag push `refs/tags/collab-v<version>`, dispatch ref/tag input, and receipt
tag. The nested dispatch `inputs` has exactly the four shown keys, repeats the
approved top-level values, and has no extras. Its `ref` is the exact short
release tag passed to GitHub.

Here `<hex>` means the exact 40- or 64-character lowercase Git revision, or a
64-character lowercase SHA-256 where named. The package entry point is
`@agenttool/collab`:

```ts
import { requestSha256 } from "@agenttool/collab";
```

If that pinned helper is unavailable, stop with a named digest-helper boundary
instead of hand-rolling a release claim. The npm tag push belongs inside the
compound `npm-release` action and must not also be claimed as
`github-branch`.

Reuse an idempotency key only for the exact same request; the relay rejects a
changed body with `idempotency_mismatch`.

Slot phases are:

```text
idle | claimed | executing | recovery_required
```

Run statuses are:

```text
claimed | executing | succeeded | failed | cancelled | uncertain |
released | recovery_required
```

Claim before seeking the final provider action. Call begin immediately before
the external mutation. Release an unbegun action that will not run. If an
executing lease expires, disconnects, or loses its host, preserve
`recovery_required` and an uncertain outcome. Inspect the provider and receipts
before calling `collab_operation_recover` with the expected version/generation,
bounded reason, and optional receipt/evidence references. An `uncertain`
recovery keeps the slot closed; `succeeded`, `failed`, or `cancelled` clears it
while preserving the history. Every observation names its observing session;
the MCP runtime supplies one stable observing `session_id` for its process
(using the bound local session where available). Callers do not mint a new ID
per observation. Bind the action when known. Never assume timer expiry means
no mutation.

Exact idempotent mutation receipts can be historical. The official client
checks lease freshness and immediately status-confirms the same complete
current slot fence before returning `claim`, `renew`, or `begin` as actionable.
If using HTTP directly, do the same before using the lease.

Status is a read-only server-time effective scan. Follow `next_after` while
`has_more` is true, then restart the next poll with the terminal
`next_after: 0` so time-only expiry remains visible. Status does not
materialize lease/event state; recovery or the next fenced mutation does, and
read authentication does not update device usage telemetry.

The room blocks cooperating duplicate claims only. A human, another tool, or
another device can still invoke a provider directly.

## GitHub review and merge

Use this order:

```text
local reviewed edit task
  → separately authorised branch push and pull request
  → observe checks and review
  → separately authorised merge to main
  → determine affected release surfaces
```

Required checks are exactly `API and protocol` and
`Data, ADDS, and SDK`. Keep other observed checks, including Python matrix and
Whitehack, separate from the required set.

The committed names are the expected policy, not proof that GitHub branch
protection has not changed. Before merge, compare them with the live protected
branch/check state and stop for reconciliation if the provider policy differs.

Do not create/edit/comment/review/merge a pull request, push/delete a branch or
tag, dispatch/rerun/cancel a workflow, approve an environment, mutate a
release, or configure a GitHub App/webhook without the separate authorization
for that act.

## npm protected release

Use this order:

```text
merged main at exact source revision
  → claim npm-release / production
  → separately authorised collab-v tag and workflow dispatch
  → protected npm-bootstrap environment approval
  → OIDC trusted publication with provenance
  → public registry and GitHub Release asset verification
  → observe receipt/provider state
  → complete room action
```

Never publish directly from a model-facing tool. Use
`.github/workflows/publish-npm.yml` and `bin/npm-release.ts`.

Treat the annotated tag push and workflow dispatch as one compound
`npm-release / production` action. Call begin before pushing the tag. If the
host disappears after the tag push or after dispatch, do not retry either step
until recovery has inspected the Git tag, workflow runs, registry, release
asset, and any receipt.

Reuse the exact `agenttool.npm-release/1` receipt:

```text
schema
package { key, name, version, path }
tag
tag_commit
source_revision
artifact { filename, size, sha1, sha256, integrity }
prepared_at
result {
  status: published | already_published_exact
  npm_tag
  registry_observed_at
  registry_tarball
}
```

Import the receipt only when its package is `@agenttool/collab`, release key
is `collab`, path is `packages/collab`, and tag is exactly the committed
`collab-v` prefix plus the receipt version.

Verify package bytes, integrity, provenance, dist-tag, and release-asset digest.
npm dist-tags and GitHub Release assets are mutable; a URL alone is not
immutable evidence.

Publishing, changing a dist-tag, deprecating/unpublishing, configuring trusted
publishing, or changing npm credentials remains separately authorised.

## Fly and Cloudflare deployment

Use this order:

```text
merged GitHub main
  → determine affected surfaces
  → claim production-deploy / production at exact revision
  → separately authorised bin/deploy.sh
  → begin before migration/Fly/Cloudflare mutation
  → verify hosted state
  → observe receipt/provider state
  → complete room action
```

Use `bin/deploy.sh` for a release-tracked production deployment. Its local
mutex cannot serialize another device, CI, or direct provider use.

Reuse the exact `agenttool-deploy-receipt/v2` fields:

```text
schema
outcome
completed_at
exit_status
source_revision
source_dirty
release_head_snapshot { remote, branch, revision, observed_at }
source_overrides { dirty, non_release_head }
external_mutation_started
phases { migrations, preflight, api, frontends }
verified_api_machines
```

The fixed receipt adapter accepts only Fly `agenttool` in `production` when the
wrapper outcome is `succeeded` and `phases.api` is exactly
`deployed_verified`. Its caller-supplied provider/environment/resource context
must match the profile, but those values are absent from the receipt and are
not provider provenance.

The adapter refuses skipped or unverified API phases and all Cloudflare Pages
or Vercel receipt imports. The receipt has only one aggregate `frontends` phase
and cannot identify `agenttool-docs`, `agenttool-dashboard`, or
`agenttool-web`. Use separately corroborated direct observations for those
surfaces.

A direct observation may name a check run, workflow run, or deployment ID.
Both direct and receipt-derived observations are attributed
`device_observed` claims, never provider-verified facts or provider-project
binding proof. The current release-room HTTP surface has no hosted provider
webhook receiver.
When Vercel is disabled, the relay rejects every new Vercel observation; an
exact historical receipt may still replay without appending state or enabling
the provider.

Corroborate the receipt with public API health and provider observations.
Receipt absence does not prove no mutation: host loss, `SIGKILL`, or receipt
write failure can interrupt recording. GitHub Deployments is not the source of
truth for Fly or Cloudflare state.

Database migration, Fly deploy/rollback/secret changes, Cloudflare Pages
upload/rollback/settings, and Codeberg mirroring remain separately authorised.

## Optional Vercel adapter

AgentTool is not a Vercel-backed repository. For another Kingdom project,
enable Vercel only after its project profile binds stable team and project IDs.
Keep preview and production distinct. Validate webhook signatures or identify
poll-derived evidence explicitly.

Treat deployment checks, ready/error events, and webhooks as observations.
Linking, creating, deploying, promoting, redeploying, cancelling, deleting,
changing domains/environment variables, and configuring a webhook remain
separately authorised.

## Evidence and privacy

Record only stable resource IDs, exact revisions, bounded native/normalized
states, receipt fields, digests, occurrence/observation time, and safe provider
links. Preserve out-of-order events rather than overwriting them.

Do not record raw logs, diffs, source bodies, prompts, transcripts,
chain-of-thought, command output, environment dumps, credentials,
secret-bearing URLs, unnecessary personal data, or absolute local paths.

Checks, reviews, receipts, provider observations, task acceptance, signed
Correspondence, and operation completion remain evidence within their
boundary. None grants external authority.

Refusal, rest, pause, uncertainty, disagreement, privacy, credit, handoff, and
repair are valid protocol states. Offer handoff rather than imposing it.
Append corrections and recovery records instead of rewriting attributed
history. Keep rights distinct from permissions.
