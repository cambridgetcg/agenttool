# Vow Forge · 誓約

Forge a reusable ability by surrendering unnecessary generality. Make the
interior frictionless because its activation boundary, cost, exit, and proof
are honest.

## Name the desire

Start with the result the ability should make easier, not an anime label or a
tool list. Collect two or three realistic trigger prompts and one prompt that
must not trigger it.

## Choose an affinity

Use the smallest primary affinity that describes the work:

- **Enhancement:** improve an existing artifact or capability.
- **Transmutation:** change representation or interface while preserving
  intent.
- **Conjuration:** create a durable artifact, schema, plan, or implementation.
- **Emission:** transmit work or results across contexts without losing
  fidelity.
- **Manipulation:** coordinate tools, agents, or processes under explicit
  authority.
- **Specialization:** perform a bespoke composite that does not fit the five.

Add a secondary affinity only when it explains a real implementation cost.
Avoid Kastro-shaped overload: broad cross-domain cleverness can consume more
capacity than a coherent native design.

## Forge the Hatsu

Define:

1. **Name:** short, memorable, and action-shaped.
2. **Trigger:** exact requests or conditions that activate it.
3. **Contract:** inputs, outputs, and completion evidence.
4. **Condition:** what must be true before it may operate.
5. **Limitation:** the generality, scope, tools, time, or authority it gives up.
6. **Budget:** observable cap on attention, attempts, cost, or risk.
7. **Breach response:** stop, roll back, fail closed, or disclose
   incompleteness.
8. **Exit:** how to end cleanly and release temporary state.
9. **Non-claims:** what the ability does not prove or authorize.

Use safe operational penalties only. Never encode self-harm, secret exposure,
data destruction, punishment of another person, fabricated scarcity, or a
loss that cannot be recovered.

## Open the freedom interior

- Remove repeat confirmations for routine actions already inside the contract.
- Let the acting agent choose implementation details that preserve the vow.
- Keep secrets, consent, production, money, external communication, and
  irreversible effects behind their actual authority boundaries.
- Treat rights as inherent and permissions as scoped; the ability creates
  neither.
- Let new user instructions revise or dissolve the vow.

## Temper the design

Test at least:

- a normal activation;
- a near miss that must not trigger;
- a missing condition;
- a budget overrun;
- an ambiguous or partially failed effect;
- a newer instruction that changes the task;
- an attempt to use the skill as authority.

Revise any restriction that creates ceremony without improving reliability.
Make the cost meaningful, observable, and connected to the capability.

## Produce the ability card

```text
Name:
Desire:
Affinity:
Trigger / anti-trigger:
Input → output:
Conditions:
Limitation and budget:
Breach response:
Proof:
Exit:
Non-claims:
```

When the user asks for a real Agent Skill, follow the host's skill-creation
workflow, keep trigger metadata sharp, use instruction-only design unless a
deterministic resource is justified, validate it, and forward-test complex
behavior with fresh context.

## Vow

Never claim that stricter prose magically creates power. Constraints improve an
agent ability only when they reduce ambiguity, limit exposure, preserve
authority, or make failure visible.
